### Slide 2 — “Agent to Agent (A2A)”

Describes the **Google A2A protocol** architecture:

**Agent Card (discovery)**
- JSON metadata: name, capabilities, skills, security schemes, supported interfaces
- Discovery via `/.well-known/agent-card.json`, registry, or manual config
- Client fetches the card before communicating

**Communication flow**
1. **Discovery** — `GET` agent card → server returns Agent Card
2. **Authentication** — parse `securitySchemes`; if `openidConnect`, get JWT from auth server (you specified **без авторизации**, so this can be skipped)
3. **`sendMessage`** — `POST /sendMessage` (with JWT in full protocol); server creates a **Task**, returns Task response
4. **`sendMessageStream`** — `POST /sendMessageStream` for long tasks; SSE events: `Task (Submitted)` → `TaskStatusUpdateEvent (Working)` → `TaskArtifactUpdateEvent` → `TaskStatusUpdateEvent (Completed)`

**Data model**
- **Message** — JSON-RPC: `Role` (user/agent), `Parts` (text, files, structured JSON)
- **Task** — central entity: ID, current status, message history, result **Artifacts**
- Long-running tasks: polling, SSE streaming, push/webhooks

**Orchestrator pattern (from slide)**
- Client/orchestrator agent selects the right remote agent by task/skill (e.g. legal vs finance agent)
